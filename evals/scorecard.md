- **短板地图**（每条 = 下一轮迭代输入）：
  1. **工程**：HF `--task text-generation` 召回的是"训模型研究员"，与"推理 serving/量化工程师"是**不同人群**——SKILL 需澄清 HF task 适合找训练/模型工程师、推理 infra 要 GitHub-first；无 GitHub token 卡住 vLLM/SGLang 贡献者深度核验；shortlist 团队集中(2 人同队)。
  2. **学术**：OpenAlex 中文消歧噪声严重(上游不可修，连 Shuai Bai 都被挂材料化学)，靠 Agent 二次核验救回=有效但成本高；search 端点匿名限流，需 DOI/work-id；**干净源 OpenReview leg 尚未做**。
  3. **GTM**：增长负责人的**个人操盘贡献难直接核**(证据多为公司层面 ARR/用户数)；单源领英+聚合站职务镜像使 2/3 候选二源偏薄；firecrawl 401(已回退 WebSearch)。
  4. **创始人**：**无专属流程**(靠通用 sifta-search 手动)；AI Coding/Agent 出圈者几乎都已成融资创始人→可招池空；独立开发者名单缺"个人职业资料+贡献深度+身份交叉"三重达标。
  5. **跨画像·隐私边界**：学术、创始人报告都贴了公开邮箱(学术署名/公司邮箱)作互链证据——虽公开，但记分卡应示范严格门：互链用主页/GitHub 链接即可，**不必输出邮箱**。需统一进 shared-gates。

- **关键变更 / 归因**：首轮建立基线。工程/学术分高，得益于本轮前已修的召回瓶颈 + Agent 判级链路(脚本召回噪声→Agent 交叉核验救回)实测有效。GTM/创始人较低=画像本身固有难点(GTM 个人证据难核、创始人可招池空+无专属流程)。
- **下一步**（按 ROI）：
  1. **P1 补 OpenReview leg**(学术干净源，直接抬学术上限、减 Agent 核验成本)。
  2. **P1 GitHub token**(资源，解锁工程 vLLM/SGLang 深度核验 + paper→PwC→contributors)。
  3. **P2 统一隐私边界**(互链不贴邮箱，写进 shared-gates)——低成本、直接消一个跨画像短板。
  4. **P2 创始人专属流程**(榜单反查 + 可招/标杆二分输出)。
  5. **P3 HF SKILL 澄清** task 适用人群(训练 vs 推理，避免工程画像方向偏差)。
