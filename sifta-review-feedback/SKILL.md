# Sifta 人工反馈

一轮 Sifta 结果之后使用。用户反馈不是总结文案，而是下一轮寻访的约束、排除项、分类和扩展种子。
必须有上一轮候选结果、反馈包或明确候选人/来源线索；没有则硬停止，要求用户提供上一轮结果。

## 执行流程

1. 读取上一轮候选人、来源地图、风险提示和用户人工反馈。
2. 把反馈整理成 `feedbackIngest` JSON：保留约束、排除项、扩展种子和分类调整。
3. 按来源拆分下一轮请求，不把 GitHub 英文 query 和 LinkedIn 中文画像混在一起。
4. 用 `sifta-cli find-people --feedback '<json>'` 分别执行每个指定来源的请求。
5. 把候选人迁移到通过、排除、顾问/推荐人、产业标杆、已知人脉或待核验线索等状态。
6. 保留上一轮风险提示、来源约束和用户明确排除项。

示例：

```bash
sifta-cli find-people \
  --query "<下一轮指定来源查询>" \
  --checkpoint "<用户原始目标>" \
  --feedback '[{"feedback":"上一轮候选人更像顾问，请继续找全职候选","constraints":["保留工程落地证据"],"exclusions":["纯论文资料"]}]' \
  --sources '["github"]'
```

## 用户可见输出

最终答复给招聘负责人看，不是反馈 JSON、命令计划或九段审查表。默认只输出 4 段决策简报：

```text
结论 → 人选和证据 → 风险和待核验 → 下一步
```

九项不是用户可见版式，只是内部防漏检查：项目简报、来源地图、候选人分桶、待核验线索、适配证明包、覆盖风险、停止条件、执行合同、下一步。它们必须折进 4 段里，不得渲染成九个顶层标题。

输出要求：

- `结论`：说明接受了哪些人工反馈、排除了哪些方向、本轮是否产生可推进人选。
- `人选和证据`：用招聘视角表格表达 `分桶 / 人选或线索 / 招聘判断 / 为什么值得聊 / 把握 / 证据来源 / 下一步`。
- `风险和待核验`：合并呈现已排除或降级的人、覆盖风险、反馈约束和执行边界。
- `下一步`：只给 1-3 个招聘动作，例如核 ownership、补第二来源、扩大同来源召回或写低压触达草稿。
- 不输出 `feedbackIngest` JSON、CLI 命令、工具调用、脚本名、events、timing、`target-count` 或使用了哪个 skill。
- 若连接器失败、未认证、返回 0 人或无法执行下一轮搜索，不编造候选人；输出修正后的寻访计划、阻塞项和下一步恢复动作。

## 反馈映射

| 用户反馈           | 下一轮约束                                                 |
| ------------------ | ---------------------------------------------------------- |
| 太学术             | 降低纯论文型资料权重；增加工程落地、系统实现或创业证据要求 |
| 更像顾问/推荐人    | 移入顾问/推荐人分类；通过学生、共同作者、前同事继续扩展    |
| 不要推荐头部创始人 | 保留为顾问、标杆或推荐入口，不作为普通全职候选人           |
| 方向对但证据弱     | 保留为待核验线索；增加个人资料、职业经历和项目证据要求     |
| 已经认识这个人     | 标记为已知人脉；作为图谱邻居种子使用                       |
| 身份不匹配         | 标记身份冲突；不要合并跨来源资料                           |

## 分来源规则

- GitHub 下一轮查询保持英文技术关键词和角色词。
- LinkedIn/GTM 下一轮查询保留用户自然语言人才画像。
- 学术图谱下一轮查询保留来源地图和个人资料核验约束。
- 多来源反馈必须拆成指定来源的独立请求。
- 保留用户明确排除项，不把已排除候选人或类型换个说法重新推荐。
- Product/GTM 场景默认中国/中文生态相关人才池优先；只看公开职业信号，不凭姓名、照片、族裔或国籍推断。
- 当反馈强调 full-time operator 时，CEO、创始人、headline-only 增长/顾问资料默认降级为顾问、推荐人、产业标杆或待核验线索，除非有明确全职可推进信号。
