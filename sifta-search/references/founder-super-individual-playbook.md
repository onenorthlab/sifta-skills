# 创始人 / 超级个体寻访手册

## 何时用

目标是"创始人型 / 超级个体 / 技术型创始人 / 有创业潜质的独立开发者"——即以**个人影响力和出圈作品**定义的人，而不是某公司里的某个职能岗。方向仍落在具体能力画像（AI Coding/Agent、infra、模型、增长…），所以**底层要并用**工程/学术/GTM 场景技能去核能力证据；本手册只加创始人画像**特有**的入口方式和判断门。

## 核心难点（先理解，别硬来）

1. **能力 ≠ 可招**。超级个体默认"都在自己做点什么"——出圈作品越强，越可能已经在创业或独立接项目。把一个在营融资创始人当"可招全职候选"是失真的。
2. **可招池天然薄**。某个热门方向里"出圈"的中文个体，往往已经转成融资创始人（作品→公司）。这不是找不到人，是这个群体的真实结构。要诚实呈现，而不是凑一张"可招候选"表。

## 找人入口：用作品/榜单反查具名人

泛搜个人无效，创始人/超级个体尤其要**从作品和榜单反查到具体的人**。下面是入口**方式**（不是穷尽名单，按目标方向自己找对应榜单）：

- 开源出圈：GitHub Trending / 高 star 项目 owner、HF 热门作者、某方向的标志性 repo 维护者。
- 资本与媒体盘点：该方向的融资盘点、媒体年度人物 / 30 Under 30 / 榜单、行业媒体（36氪、智源、机器之心等）的创始人专访。
- 产品出圈：某个刷屏产品/Agent 的创始团队、独立开发者社区（如中文独立开发者盘点）的作者。

从榜单/作品拿到人名后，再回到能力画像 skill 核验其贡献深度与身份。

## 强制判断门（创始人画像特有）

- **在营融资创始人 → 默认降级为顾问 / 引荐 / 行业标杆池，不作可招全职候选。** 他们的价值是标杆对照、行业顾问、引荐入口，不是招募对象。除非有明确信号（如已宣布退出/公司已关停）才另议。
- **可招 vs 标杆，必须二分显式标注。** 每个人明确写清是"可招候选"还是"顾问/标杆/引荐"，不要含糊地混在一张推荐表里让招聘方误判。
- **可招池薄时诚实报缺口。** 若达标的都是在营创始人（=标杆），就直说"本轮可招候选为 0，均为标杆"，并说明原因（出圈者多已创业）；给出放宽画像的下一步（如"有出圈开源项目、但尚未融资开公司的独立开发者"再跑一轮），而不是把创始人包装成可招。

## 身份与隐私门

- 升级门槛：出圈作品 + **≥2 条公开互链**（GitHub / 个人主页 / 媒体专访 / 产品或公司页）一致。单一线索只能进其他线索。
- 公开职业信息（含公开职业/公司邮箱）可输出并标来源；不推断求职意愿——"可招/标杆"是招聘边界分类，不是当事人意愿。隐私门同 [shared-gates.md](shared-gates.md) §3。

## 与跨源融合的关系

超级个体常跨画像（研究型工程师、infra 创业 CTO）。同轮为同一画像跑了多个来源时，用 `scripts/merge-proposals.mjs` 按强公开身份锚保守合并；`crossSourceConfirmed` 的人正是这种跨画像超级个体，优先核验——但跨源确认是身份可信度增强，不自动等于可招。
