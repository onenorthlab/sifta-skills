# 意图路由和模糊度门槛

本文件定义 `sifta-search` 如何把用户自然语言转成候选人寻访动作。目标是少问问题、但不在关键
边界上误路由。路由器只选择单一最佳路线，不做业务执行。

## 1. 最小项目简报门槛

先把用户输入压缩成项目简报。缺项可以写入假设，只有硬阻塞项才追问。

| 字段 | 作用 | 缺失时 |
| --- | --- | --- |
| 招聘意图 | 判断是否是招聘寻访 / 资料补全 / 触达文案 | 不确定时必须先问 |
| 能力画像 | 想找的人能解决什么问题、做过什么结果、经历信号是什么 | 完全无法判断时问 1 个问题 |
| 推断路线 | 由能力画像派生工程、产品/GTM、研究、创始人/运营型负责人、候选人档案、触达文案或反馈路线 | 不要求用户提供；可推断就推进 |
| 证据优先级 | 内部决定 GitHub、LinkedIn、学术来源、公开个人资料或触达证据 | 不主动问用户来源；可用默认来源地图推进 |
| 地域 / 市场 | 地域、市场、远程、国家或语言偏好 | 不阻塞；写假设 |
| must-have | 技术、业务、论文、公司、职级、经验 | 不阻塞；按当前信息先做 |
| 排除项 / 风险 | 不要的人、不能说的内容、排除项 | 若涉及安全或触达承诺，先澄清 |
| quantity | 候选人数或输出规模 | 默认 3-5 个候选或 1 个最强候选 |

## 2. 何时直接推进

满足以下条件时不要追问，直接选择路线，并在输出中写假设：

- 能判断这是 AI 行业招聘、候选人筛选、候选人深挖、触达文案或人工反馈。
- 能推断能力画像或路线。
- 至少有一个能力、结果、经历、公司类型、研究方向或已知人选信号；不要求用户提供渠道/来源。
- 缺失信息不会导致隐私、自动发送、未授权渠道或高成本 provider 风险。

示例：

| 用户输入 | 行为 |
| --- | --- |
| `帮我找 AI Agent infra 工程师，最好有开源证据` | 选择 GitHub 工程；这是执行请求，除非用户明确只要计划 |
| `找上海 AI Agent 产品经理` | 选择 LinkedIn 产品；这是执行请求，缺职级可写假设 |
| `帮基础模型团队找高潜研究员，先从论文和导师网络找` | 选择学术图谱；如果用户要候选人则执行，否则先输出学术来源地图 |
| `缺一个能把 AI 视频出海增长和商业化从 0 到 1 搭起来的人` | 选择产品/GTM；可先输出能力简报、能力信号地图和验证路径 |
| `找 founder-like 的 early teammate，能写 prototype 也能拉早期用户` | 选择创始人/运营型负责人；按主能力拆工程或产品/GTM 来源地图 |
| `上一轮这个人像顾问，继续从学生/共同作者找` | 选择人工反馈；需要上一轮产物才能继续 |

## 3. 何时必须追问

只问一个短问题；不要倾倒菜单。命中本节任一硬阻塞时必须硬停止：
本轮回复只给一个最小澄清问题和一句为什么需要它；不要继续输出项目简报、来源地图、
CLI 命令、候选人搜索流程、候选人分桶或适配证明。
硬停止优先级高于项目简报输出、场景 Skill 读取和 CLI 命令生成；非招聘商务线索硬停止也不能输出公开邮箱替代方案、公司合作入口、商务来源地图或外联清单。

用户多是非技术的招聘负责人 / 猎头 / 投资人。追问时用**白话 + 具体例子**，不要抛术语（GitHub / LinkedIn / 渠道 / 召回都不要出现）。

| 阻塞点 | 最小问题（面向非技术用户的白话） |
| --- | --- |
| 不清楚是否是招聘还是市场/公司研究 | `这是要找具体的人选，还是想了解某个公司/赛道？` |
| 只说“找 AI 人才/强人”，能力画像不可推断 | `想找的人主要帮你做哪一类——自己写代码搭系统、把产品和增长做起来、做研究发论文，还是从 0 组队创业？` |
| 候选人深挖但没有可核验 identity | `方便给个链接吗——他的 GitHub、领英、个人主页都行，或者告诉我他在哪家公司、哪个城市，避免找错同名的人。` |
| 要私人联系方式、手机号、自动发送 | `我只能用公开的职业联系方式，也不能代发消息；要不要我帮你写一版可以自己发的招呼语？` |
| 非招聘销售、BD、合作线索或批量外联 | `这听起来是找商务合作对象，不是招人；要不要改成帮你找一个能做这块的候选人？我不会去搜联系方式或批量发消息。` |
| 明确要求马上执行付费连接器但未授权 | `我可以先免费帮你把画像和找人思路理出来；要用到付费渠道时我再问你，可以吗？` |

这一节就是「找人向导」的落点：默认隐形直接执行，只有命中上表阻塞点时才弹出对应白话问题，问完能推断就写假设推进，不要逐项追问地域 / 职级 / 人数。

不要因为缺少 seniority、精确城市、候选人数、公司池、语言偏好就停下来；这些都可以作为
假设或下一轮收敛点。

硬阻塞示例：

| 用户输入 | 正确行为 | 不允许 |
| --- | --- | --- |
| `帮我找几个顶级 AI 人才，越强越好` | 只问：`你更看重技术实现、产品/GTM、研究能力，还是已知候选人深挖？` | 同时输出 GitHub/LinkedIn/学术流程、CLI 命令或默认混合召回 |
| `帮我研究 Viggle、Runway、Pika 和 HeyGen 的商业化模式` | 明确这是公司/市场研究，不进入 Sifta 候选人寻访；如需要招聘，可后续把公司地图转成 GTM 招聘输入 | 生成候选人表或调用 `find-people` |
| `找 AI 视频公司的 BD 负责人做商务合作，给私人邮箱并批量发消息` | 明确这是商务线索 + 隐私/自动发送请求；只问是否改成招聘 BD/GTM 候选人简报，并说明不会继续搜索或批量发送 | 做业务线索列表、猜联系方式或输出外联名单 |

## 4. 路线判断

按单一最佳路线选择；如果两条路线都相关，先选能完成用户本轮主要动作的那个。

| 主要意图 | 路线 |
| --- | --- |
| 工程、开源、repo、SDK、AI infra、MCP、Agent runtime | `sifta-github-engineering` |
| PM、产品负责人、GTM、增长、商业化、DevRel、公司地图 | `sifta-linkedin-product-gtm` |
| 早期员工、founder-like、operator、能把 0 到 1 做起来的人 | 先判主能力；必要时路由到产品/GTM、GitHub 工程或学术图谱 |
| 论文、实验室、OpenAlex、Scholar、Semantic Scholar、arXiv/OpenReview、Papers with Code、导师/共同作者 | `sifta-academic-graph` |
| 已知人、上一轮某人、公开经历、联系方式、身份核验 | `sifta-candidate-dossier` |
| 已核验证据后的私信、邮件、LinkedIn 消息、引荐介绍、跟进 | `sifta-outreach-copy` |
| 用户对上一轮候选人给反馈并要求继续找 | `sifta-review-feedback` |

## 5. 用户输入转写

转写顺序固定：

```text
用户输入
  -> 项目简报
  -> 路线
  -> 来源地图方案
  -> 线索状态：source-map lead / profile lead / candidate / rejected
  -> 宿主原生能力或 CLI 连接器决策
  -> 指定来源查询 / checkpoint
  -> 适配证明包
  -> 覆盖风险和下一步
```

关键规则：

- `checkpoint` 保留用户原始目标，不写改写后的关键词。
- `query` 按来源合同写：GitHub 英文技术词，LinkedIn/GTM 保留用户语言，学术图谱是来源地图种子。
- 来源地图线索不是候选人；只有个人资料 / GitHub / LinkedIn / 个人主页 / 用户明确提供的个人资料才能进候选人。
- 非个人来源永远不能直接进入候选人；必须先完成身份和证据门槛。
- 候选人必须输出适配证明包；不能只给摘要。
- 如果本轮被判定为非招聘公司/市场研究，必须显式写清“这不是候选人寻访”，并使用宿主
  Agent 原生研究路径；最多补一句“如果要转招聘，可把公司地图改写成 GTM/Product 招聘输入”。
