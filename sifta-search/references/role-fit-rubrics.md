# 角色级适配证明标准

本文件定义不同人才画像的强证据。它补充通用适配证明包，避免所有角色都用同一套“头衔 + 简介”判断。

## 1. 通用规则

| 项   | 要求                                                                                                    |
| ---- | ------------------------------------------------------------------------------------------------------- |
| 身份 | 有个人资料、个人主页、GitHub、LinkedIn、Scholar/lab bio、X 或用户给定 profile；跨渠道合并必须有公开证据 |
| 能力 | 至少一个与用户需求直接相关的公开证据                                                                    |
| 角色 | 按主要证据分类，不按用户 query 里的 AI/Agent 词硬归类                                                   |
| 地域 | 默认中国/中文生态优先；只看公开职业信号，不猜身份属性                                                   |
| 风险 | 每个人都写还要确认什么；强候选也不例外                                                                  |

## 2. 工程师 / 开源候选

| 证据强度 | 例子                                                                                            | 输出                   |
| -------- | ----------------------------------------------------------------------------------------------- | ---------------------- |
| 强       | 核心 repo owner/maintainer、长期 PR、架构性 issue、package 维护、官方实现、技术文章与 repo 互证 | 候选人分桶：可优先聊   |
| 中       | 单个强 repo、公司/主页证明角色，但贡献深度未完全核实                                            | 候选人分桶：建议先核实 |
| 弱       | star/fork、awesome list、组织仓库 owner、关键词命中                                             | 待核验线索             |

必须确认：这个人是否亲自做了目标技术方向的核心实现，而不是只是收藏、转发或挂组织名。

## 3. 研究员 / 研究工程师

| 证据强度 | 例子                                                                                        | 输出                 |
| -------- | ------------------------------------------------------------------------------------------- | -------------------- |
| 强       | 一作/共一作 + 个人主页/lab bio + official code/GitHub/项目页，或公司研究岗位与论文/实现互证 | 候选人分桶：可优先聊 |
| 中       | 论文贡献明确，但个人资料或职业阶段缺一项                                                    | 待确认人选           |
| 弱       | 只出现在论文作者列表、导师页、共同作者图谱或引用网络                                        | 待核验线索           |

PI、导师、头部科学家通常先进顾问/推荐人/产业标杆，除非用户明确找科学家全职且有可招聘性证据。

## 4. 产品经理 / 平台负责人

| 证据强度 | 例子                                                                              | 输出                 |
| -------- | --------------------------------------------------------------------------------- | -------------------- |
| 强       | 职业资料显示负责 AI 产品/平台/开发者工具，且有发布、roadmap、产品线或用户问题证据 | 候选人分桶：可优先聊 |
| 中       | 有产品 title 和 AI 公司经历，但 ownership 不清                                    | 待确认人选           |
| 弱       | 只在 AI 公司任职，无法证明负责相关产品                                            | 待核验线索           |

不要因为候选人所在公司做大模型或 Agent，就把所有 PM 当作目标产品人。

## 5. GTM / 增长 / DevRel / 市场

| 证据强度 | 例子                                                                                            | 输出                       |
| -------- | ----------------------------------------------------------------------------------------------- | -------------------------- |
| 强       | 明确负责开发者增长、商业化、出海、社区、生态合作、DevRel 或产品营销，并有公开项目/活动/增长结果 | 候选人分桶：可优先聊       |
| 中       | 职能匹配但结果证据不足                                                                          | 待确认人选                 |
| 弱       | 商务合作对象、KOL、销售名单、媒体联系人、公司渠道页                                             | 不作为候选人；必要时硬停止 |

partnerships 只在它是候选人的职业职能时成立；不能把合作伙伴名单当招聘候选人。

## 6. 创始人型 / operator / 早期员工

先区分用途，不要把所有强人放进候选人分桶：

| 推荐级别            | 条件                                                                          | 用途                         |
| ------------------- | ----------------------------------------------------------------------------- | ---------------------------- |
| 候选                | 0 到 1 产品/技术/增长证据强，个人资料可核验，且没有明显深度绑定自己公司的信号 | strong / soft shortlist      |
| benchmark           | 证据很强但明显不可招，如已大额融资、团队很大或公开长期投入自己公司            | 产业标杆，校准画像，不当候选 |
| advisor / reference | 影响力强但不亲自 build，或更适合作推荐人/顾问入口                             | 推荐人、顾问、关系扩展       |
| lead                | 方向对但主导权、产品、用户、可招性证据不足                                    | 待核验线索                   |

| 证据强度 | 例子                                                                                      | 输出                    |
| -------- | ----------------------------------------------------------------------------------------- | ----------------------- |
| 强       | 0 到 1 产品/技术/增长组合证据，公开产品、团队、用户或融资/营收/分发记录，且个人资料可核验 | 创始人级候选或顾问      |
| 中       | 能证明做过早期产品或技术，但可加入性未知                                                  | 顾问/推荐人或待确认人选 |
| 弱       | 只有 founder title、媒体报道或公司介绍                                                    | 产业标杆 / 待核验线索   |

要先判断用户是真的要招 founder-like teammate，还是要找标杆学习对象。

## 7. 超级个体 / 独立开发者

| 证据强度 | 例子                                                          | 输出                               |
| -------- | ------------------------------------------------------------- | ---------------------------------- |
| 强       | 独立产品 + 公开用户/收入/分发/社区反馈 + 技术或产品贡献可核验 | 候选人分桶：可优先聊或创始人级候选 |
| 中       | 有产品或内容影响力，但技术/商业化贡献缺交叉验证               | 待确认人选                         |
| 弱       | 只有内容账号、课程、转发或榜单                                | 待核验线索                         |

独立开发者不是“粉丝多的人”。必须证明他做出了东西，且和用户目标相关。

## 8. 用户可见翻译

内部可以按强/中/弱判断；对用户只写：

- `把握：高`：公开证据扎实，可以优先聊。
- `把握：中`：方向对，建议先核实一两点。
- `待核验线索`：还不能算人选，只说明为什么值得继续确认。
