# 输出、质量门和失败恢复

本文件给 `sifta-search` 主 Skill 提供详细执行口径。主 Skill 只负责分流和高优先级规则；

## 0. 线索 -> 候选人状态机

| 状态            | 说明                                                       | 升级门槛                                                                     |
| --------------- | ---------------------------------------------------------- | ---------------------------------------------------------------------------- |
| `sourceMapLead` | 论文、仓库、公司、实验室、项目、数据集、共同作者等入口线索 | 找到个人资料、GitHub、LinkedIn、个人主页、实验室简介或用户明确提供的个人资料 |
| `profileLead`   | 已有个人资料，但适配或身份交叉证据不足                     | 身份置信度至少为中等，并有与用户要求相关的公开证据                           |
| `candidate`     | 可进入推荐人选的人                                         | 已完成证据评级、推荐级别明确、适配证明包完整                                 |
| `rejected`      | 身份冲突、隐私、非公开数据、无关或证据不可追溯             | 写明拒绝/排除原因，不为凑数保留                                              |

论文 / 仓库 / 公司 / 实验室线索不能直接变候选人。弱结果优先回到来源地图或个人资料核验，
不要临时补一个看似更强的人填候选人列表。

## 1. 结构字段口径

结构化字段要跟候选人的主要公开证据一致，不要被用户目标里的技术词或单个头衔关键词带偏。
先判断这名候选人为什么被召回，再写 `functionCategory`、`careerStage` 和人才池。
不同角色的强证据标准见 [role-fit-rubrics.md](role-fit-rubrics.md)。

| 主要证据                                | `functionCategory` 示例              | 注意事项                                                         |
| --------------------------------------- | ------------------------------------ | ---------------------------------------------------------------- |
| 代码、模型、infra、SDK、开源贡献        | `Agent/LLM工程`、`WAM/VLA模型`       | 只有公开证据证明主要做工程实现时才归工程类                       |
| 产品规划、PM、roadmap、平台产品         | `AI产品/平台`                        | 不要只因为产品涉及 Agent/LLM/大模型就归工程类                    |
| 增长、市场、商业化、DevRel              | `GTM/增长/DevRel`                    | 不要只因为所在公司是 AI 产品公司就归产品岗                       |
| 战略规划、商业分析、投资                | `战略/CEO Office/商业分析`           | 需要职业资料或公开经历支撑                                       |
| 论文、实验室、PI、产业科学家            | `科学顾问资源网络` 或具体研究类      | 强学术人物可能是顾问、推荐人、产业标杆，不一定进入全职推荐人选 |
| 年轻高潜、博士后期/实习、竞赛和论文信号 | `基础模型研究/研究工程` 或具体研究类 | 必须同时核验个人资料、贡献强度和公开实现或职业阶段               |
| founder / co-founder / C-level          | 由主要职能决定                       | 创业/高管身份是可用性和职级信号，不是自动分类规则                |

提交结构字段前做一致性检查：

- 工程证据为空，且主要公开证据是产品规划、PM、产品负责人、平台产品或应用产品时，应填
  `AI产品/平台`。
- 主要证据是增长、市场、商业化、开发者社区、partnerships、DevRel 或出海时，应填
  `GTM/增长/DevRel`。
- 只有论文作者但缺少职业 / 个人资料线索时，说明是研究线索或顾问入口，不包装成完整全职候选人。
- 学术图谱通道下，年轻、博士、顶会论文或竞赛奖项都只是入口信号；没有个人
  资料、公开实现、项目贡献或职业阶段证据时，只能进入来源地图或待核验线索。
- 学术图谱通道下，如果方案没有说明 OpenAlex、Google Scholar、Semantic Scholar
  或同级广泛图谱 / 广泛召回来源是否已检查，必须写还要确认；Google Scholar
  不应被写成官方 API 连接器。

## 2. 候选人输出格式

用户是招聘负责人、猎头、投资人，不是工程师。**§0 的状态机、§3 的证据等级 A/B/C、`functionCategory`、`priority`、`checkpoint`、`sourceFamily` 等只在模型脑内推理用，绝不直接出现在给用户的回复里**；渲染时按 [templates/final-report.md](../templates/final-report.md) 的白话映射翻译。

最终回答必须是 Markdown，不是 JSON、纯文本字段块或工程汇报。默认像给招聘负责人看的报告，不写「已完成 / 验证 / 运行了 / 用了哪个 skill / 调了哪个命令 / target-count / bash / code block / events / timing」。这些过程信息只进 eval artifact、trace 或内部记录；除非用户明确问「怎么跑的」。

完整渲染模板与白话映射见 [templates/final-report.md](../templates/final-report.md)。用户可见结构默认是：
**结论 → 人选和证据 → 其他线索 / 需要确认 → 下一步**。
0 人、provider failure 或 privacy hard-stop 时，不输出候选人表，改用 **结论 → 原因 → 下一步**。
不要把内部检查项直接渲染成九个大标题，也不要让招聘负责人读工程审计表。项目简报、来源地图、候选人分层、待核验线索、适配证明、覆盖缺口、停止原因、执行边界和下一步只是内部防漏检查，必须折进上述短报告；没有新增信息量的检查项不单独展示。
不要使用「本轮目标 / 找人来源 / 推荐名单 / 匹配依据 / 本轮覆盖缺口」这组旧标题，也不要使用英文标题。

格式要求：

- 每行一个人选；「为什么值得聊」控制在 30-50 字。
- 链接用 `[主页](url)`、`[领英](url)`、`[GitHub](url)`，不输出裸 URL，不写 `github.com/...` 等渠道名当卖点。
- 如果 helper 或草稿已经输出 `[资料](url)` / `[主页](url)` / `[GitHub](url)`，最终报告必须保留 Markdown 链接，不能改成裸 URL。人选名字本身优先做成链接，不要只在最后一列放渠道链接。
- 每位人选必须同时有「为什么值得聊」和「还要确认」，否则不算完整交付。
- 推荐人选必须用招聘视角列：`人选 / 招聘判断 / 为什么值得聊 / 相关作品或证据 / 还要确认 / 下一步`。不要把 repo、PR、commit、star 或 topic 当成推荐理由本身；它们只是公开证据。
- `招聘判断` 只能写推进动作和定位，例如“可优先聊”“建议先核实”“适合顾问/推荐人”，不能写技术能力清单。
- `为什么值得聊` 只保留最能解释招聘价值的一句话。技术细节最多出现一个，超过一个就放进适配证明包。
- `下一步` 必须是招聘动作：补职业资料、核 ownership、低压触达、转顾问/推荐人、排除；不要写成技术调研 checklist。
- 弱线索、来源缺口、本轮停止原因和隐私边界只写进「其他线索 / 需要确认 / 原因 / 下一步」中，不默认渲染成内部标签表。
- 执行类报告需要说明默认地域/市场时，写成一句白话：`默认中国/中文生态优先只看公开职业信号，不凭姓名、照片、族裔或国籍推断`。
- 隐私和触达边界只在相关场景出现：不查询私人邮箱、手机号、非公开联系方式，不自动发送消息或提交表单。
- 默认中国/中文生态人才池未被用户放宽时，缺公开中国/中文生态职业信号的人不能进入推荐人选，只能放入「其他线索 / 全球备选 / 待补证入口」。
- 「把握」可在内部使用高 / 中 / 低，但不要成为默认表头；线索类只能进「其他线索」，不能混进推荐人选。
- 不使用 `候选人：`、`画像：`、`来源：` 这种逐条字段块格式。
- 来源失败、额度不足或认证失败时，只说它如何影响这轮找人的覆盖面和下一步，不写命令、参数、耗时、调用事件或用了哪个 skill。

常见失败样式：推荐人选表像技术尽调报告，堆 `sandbox / MCP / multi-session / message bus / permission / commit`。这类内容即使真实，也只能做支撑证据；报告必须先告诉招聘负责人“为什么现在值得约这个人”。

## 3. 适配证明包

适配证明包采用 `requirement -> evidence -> source -> confidence -> weakness -> next action`。
详细标准见 [fit-proof-packet.md](fit-proof-packet.md)。

证据等级：

| 等级   | 说明                                              | 输出口径                         |
| ------ | ------------------------------------------------- | -------------------------------- |
| A      | 个人资料与目标证据直接匹配，来源可追溯            | 可进入推荐人选                   |
| B      | 强来源线索或单一个人资料证据，缺交叉验证          | 建议先核实的人选，必须写还要确认 |
| C      | 仓库 owner、论文作者、公司/实验室成员或关键词命中 | 来源地图线索或待核验线索         |
| Reject | 身份冲突、非公开信息、无关或不可追溯              | 不进推荐人选                     |

## 4. 结果解释

优先使用这些结构解释结果：

- `people`
- `searchStrategy`
- `sourceMap`
- `evidenceLog`
- `crmExport`
- `warnings`

不要只看 `summaryMarkdown`。不要编造 JSON 中不存在的字段。不要编造邮箱、电话、薪资、
搬迁意愿、在职状态或私人联系方式。不要把 `hireable=true`、`Open to new opportunities`、
`open to work`、profile bio 或状态标记写成“可招聘性”或“求职意愿”结论；最多写“公开 profile
有开放机会相关表述，触达前必须确认当前方向和是否开放外部机会”。

除非 Sifta 返回同人提示 (same-person hint)，或有明确公开证据，否则不要断言跨渠道个人资料是同一个人；
不确定时写成“可能匹配”。

## 5. 覆盖缺口和还要确认

必须把 API 返回的 warnings 翻译成招聘者能用的“还要确认 / 原因 / 下一步”，尤其是：

- provider/network 失败。
- 指定来源没有执行或没有返回候选人。
- `priority=C` 或 `evidenceStatus=缺职业工程证据`。
- 仓库回退返回仓库 owner 线索。
- 来源地图线索被降级，或缺少个人资料核验。
- 项目简报里的目标缺失但用假设推进。
- 候选人分类不确定。
- 证据只来自单一个人资料，缺少交叉验证。
- 默认地域/市场生效时，必须写明它不是姓名、照片、外貌、口音、族裔或国籍推断；只看公开职业信号。
- 用户要求不走某类连接器、来源额度不足或认证失败时，只写对覆盖面的影响和可选补救；不要默认写“未使用 / 不改用 Sifta CLI”、具体命令或认证过程。

仓库回退的口径：

- 如果只有仓库命中，缺少个人资料、公司/经历或身份信号，只能作为来源地图入口。
- 如果有明确个人资料、公司/经历信号和强开源项目证据，可以作为建议先核实的人选进入推荐人选。
- 即使进入推荐人选，也必须写清楚还要确认：人工验证仓库 owner 是否为候选人本人。

学术图谱回退的口径：

- 论文 / 实验室 / 导师 / 共同作者 / 竞赛 / 项目只能解释从哪里继续找人。
- 一作、共一作、竞赛奖项或博士阶段可以提高优先级，但不能替代身份、职业阶段和工程/研究贡献核验。
- PI、导师、通讯作者或头部科学家优先进入顾问/推荐人/产业标杆池；除非用户明确找顾问或科学家全职，并且有个人资料证据支持。
- 如果只找到论文和机构，不输出完整候选人表；应输出找人入口、其他线索和下一步个人资料核验 / 补全动作。

## 6. 失败恢复

命令参数或结构变化：

1. 运行 `sifta-cli tools`。
2. 找到 `find_people` 结构。
3. 根据当前结构重建命令。
4. 使用明确 CLI 命令重试。

以上是内部恢复动作。最终用户报告只呈现恢复后的候选人结果、还要确认和下一步；不要把重试命令、参数变化或 schema 检查过程写进用户答复。

无候选：

- 不要断言不存在这类候选人。
- 说明“这次搜索没有返回候选人”。
- 给出具体调整：放宽头衔、去掉地点、切换来源、补充公司/领域线索，或改走研究来源地图。

弱结果：

- 说明是“弱线索 / 待复核”，不能作为完成交付。
- 优先在同一来源下改写查询画像、扩大召回、补充明确公司或项目线索，或放入来源地图。
- 可以使用宿主 Agent 原生搜索补证据或扩展来源地图，但不能绕过 Sifta 质量门，临时找一个
  看似更强的人填候选人列表。

质量问题归因：

| 现象                               | 优先归因            | 修复方向                                             |
| ---------------------------------- | ------------------- | ---------------------------------------------------- |
| 渠道输入不符合按来源定制的查询合同 | Skill / 查询合同    | GitHub 改英文技术查询；LinkedIn/GTM 保留用户语言画像 |
| 未授权调用 X                       | 来源 / API 可见性   | 保留用户来源约束，检查 Public API 来源选择           |
| X-only 弱线索进入推荐候选人表       | 证据升级门槛        | 降级为待核验线索；补中国/中文生态信号、职业资料、作品或身份核验 |
| 网页/论文线索进入候选人            | 来源合同            | 修 Public API 或结果过滤                             |
| provider `fetch failed`            | Provider / 网络     | 写 warnings，不硬编码兜底候选人                      |
| 候选人质量弱但输入正确             | 覆盖不足 / 查询模式 | 调整查询、补来源地图、人工审查                       |
| 结构字段缺失                       | API 合同            | 修结构化候选人 / CRM 导出                            |

## 7. 更新检查

Sifta CLI 的 JSON 输出可能包含 `_notice.update`：

```json
{ "_notice": { "update": { "cli": "0.0.3" } } }
```

不要影响当前候选人结果解析；先完成用户当前寻访请求。版本更新、更新命令和重启提醒属于工具维护信息，不默认放进候选人报告；只有用户询问工具状态、排障或升级方式时再说明。
