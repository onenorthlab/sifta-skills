# 匹配依据

本文件定义如何证明候选人符合用户需求。Sifta 的输出必须让用户知道“为什么这个人值得看”，而不是只看到姓名和个人资料。

复杂画像或多角色寻访时，先读 [role-fit-rubrics.md](role-fit-rubrics.md)，再把对应角色的强证据落到本文件的通用证明包。

## 1. 固定结构

每个候选人必须能落到这个证明包：

| 字段        | 含义                                                                |
| ----------- | ------------------------------------------------------------------- |
| 状态        | `待确认线索` / `profile lead` / `verified candidate` / `rejected`   |
| 要求        | 用户需求或转写后的筛选条件                                          |
| 证据        | 公开证据，必须能指向来源                                            |
| 来源        | GitHub / LinkedIn / paper / homepage / repo / talk / company bio 等 |
| 置信度      | 身份、适配度、证据三类置信度                                        |
| 风险 / 弱项 | 未证明、不满足或需要人工核验的点                                    |
| 下一步      | 深挖、触达、继续扩展、转顾问/推荐人、排除                           |

## 2. 证据质量等级

| 等级   | 说明                                                      | 可作为强候选人吗                             |
| ------ | --------------------------------------------------------- | -------------------------------------------- |
| A      | 个人资料与目标证据直接匹配，且来源可追溯                  | 可以                                         |
| B      | 有强找人来源线索或单一个人资料证据，但缺少交叉验证        | 可以作为建议先核实的人选，必须写本轮覆盖缺口 |
| C      | 仓库 owner、论文作者、公司/实验室成员、关键词命中等弱线索 | 只能进找人来源或待确认线索                   |
| Reject | 身份冲突、非公开信息、与岗位无关、证据无法追溯            | 不进推荐名单                                 |

C 级只能作为线索；B 级可进入建议先核实的人选；A 级才可作为强推荐。隐私、自动发送、
无个人资料、身份冲突或非公开数据是硬边界，直接排除或硬停止。

## 3. 接受门

候选人进入推荐名单前必须满足：

- 身份置信度至少为中等，或明确写成“待确认人选”。
- 至少一个与用户要求直接相关的公开证据。
- 证据要符合对应角色标准：工程看实现/维护，研究看论文与实现/身份，产品看 ownership，GTM 看增长/社区/商业化结果，founder/独立开发者看 0 到 1 作品和分发。
- 来源链接或来源描述可追溯。
- 风险 / 弱项不为空；即使强候选也写未验证项。
- 不输出私人联系方式猜测、薪资、求职意愿、搬迁意愿、签证、入职时间。

学术图谱中的论文、实验室、导师、共同作者、竞赛和项目线索必须先停在找人来源；
只有找到个人主页、GitHub、LinkedIn、Scholar 个人资料、实验室简介或用户给出的个人资料后，才可进入推荐名单。

## 4. 输出模板

§1–§3 的状态、证据等级、置信度是**模型内部推理**用的，决定一个人能不能进推荐名单、把握标多高。**渲染给用户时不要把这些术语写出来**，按 [templates/final-report.md](../templates/final-report.md) 翻译成白话：

- 证据等级 A → 「把握：高」；B → 「把握：中（建议先核实）」；C → 进「待确认线索」。
- 「为什么值得聊」= 把「要求 → 证据 → 来源」浓缩成一句招聘负责人能懂的话（点出和岗位最相关的那条公开证据）。
- 「还要确认」= 把「风险 / 弱项 / 缺的交叉验证」写成用户能行动的一句话（如「是否在看新机会」「确认是不是同一个人」）。
- 推荐名单里的技术证据只能作为一句支撑，不要把多个 repo 功能、commit 主题或技术组件堆进「招聘判断」。长证据放到匹配依据，用户先看到的是推进动作和顾虑。

也就是说：内部跑完整的 `要求 → 证据 → 来源 → 置信度 → 风险 → 下一步`，对外只呈现「为什么值得聊 + 把握 + 还要确认」。完整用户可见模板见 `templates/final-report.md`。

## 5. 常见失败

| 失败                        | 处理                                    |
| --------------------------- | --------------------------------------- |
| 只有摘要，没有证据/来源     | 不算完成；补匹配依据                    |
| 论文作者直接当候选人        | 降级为找人来源线索，继续找个人资料      |
| 仓库回退被强推              | 降级为 B/C，保留本轮覆盖缺口            |
| LinkedIn 候选人只有头衔命中 | 写适配度低/中，要求补产品/增长/工程证据 |
| 触达文案没有个性化证据      | 不能发送；退回补证据或写通用草稿        |
