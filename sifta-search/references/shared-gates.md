# 共享执行门

本文件承接所有 Sifta 场景技能的共同规则。场景 `SKILL.md` 只写差异化流程和质量门；遇到冲突时，以具体场景技能和最终报告模板为准。

## 1. 执行、计划和硬停止

| 类型 | 触发 | 动作 |
| --- | --- | --- |
| 默认执行 | 用户要找人、推荐候选人、给名单、跑一轮，且能推断能力画像 | 进入最匹配的场景技能；缺地域、职级、数量不阻塞，写入假设后推进 |
| 只给计划 | 用户明确说先规划、怎么找、来源地图、别搜索、只要方法 | 只输出找人入口、升级门槛、风险和下一步；不调用 CLI、浏览器、网页搜索或 helper |
| 硬停止 | 不清楚是否招聘、能力画像不可推断、要求私人联系方式、自动发送、批量外联、销售/合作线索 | 不搜索；只问一个必要澄清问题，或说明可改成招聘候选人 brief |

不要问用户“从什么渠道找”。渠道由 Sifta 根据能力画像内部决定。

## 2. 默认地域和公开信号

用户未指定地区、远程、全球或海外市场时，默认 `中国/中文生态相关人才池优先`。这只是招聘寻访的来源优先级和候选人升级门槛：

- 只看公开职业资料里的中国大陆、港澳台、中文教育/工作/社区、中国市场或中国相关机构/公司信号。
- 不凭姓名、照片、外貌、口音、族裔、国籍或猜测判断。
- 缺公开中国/中文生态职业信号的人只能放入其他线索、全球备选、产业标杆或待核验对象；用户明确放宽全球人才池时除外。

## 3. 公开信息和联系方式

只使用候选人本人或机构公开展示的职业资料：GitHub、LinkedIn、X 公开 profile、个人主页、论文页、实验室页、项目页、package registry、公司 bio、公开通讯邮箱或公开职业邮箱。

- 候选人本人公开的 email、主页、profile 或论文通讯邮箱可以输出给招聘负责人，但必须标来源。
- 没找到公开联系方式时写“未找到公开职业联系方式”，不要推断邮箱格式、手机号、住址、家庭信息或 data broker 结果。
- 最终报告可以给出公开职业联系方式，但下一步必须体现人工核验 / 人工触达边界；不自动发送消息，不使用私人或未公开联系方式。
- GitHub API 当前认证上下文未返回 `email` 时，只能说明“当前 GitHub API context 未返回 profile email / 本轮来源未找到公开 email”；不要断言候选人没有公开邮箱。
- `hireable=true`、`Open to new opportunities`、`open to work`、profile bio 或状态标记不等于求职意愿；最多写成触达前待确认事项。

## 4. 工具、helper 和 Web 保存

Sifta CLI/API 是可选命令/连接器层，不是通用搜索替代品。GitHub、学术或公开资料核验可以优先使用宿主 Agent、GitHub MCP、`gh`、浏览器或原生搜索；LinkedIn/Product/GTM、反馈闭环、统一 JSON、trace 或 Web 回看需要连接器时再用 CLI/API。

- GitHub：使用英文技术关键词和角色词；用户原始目标、默认地域、排除项和风险放进 `checkpoint`、`feedback` 或最终核验。
- LinkedIn / GTM / 产品岗：使用用户语言的自然画像；保留岗位、职能、城市、公司、市场和默认中国/中文生态偏置。
- 用户显式指定来源后，所有重试和替代命令必须保留相同 `--sources`。
- 网页 / 论文 / 实验室 / 公司 / 仓库 / 数据集线索不能直接变成候选人；无个人资料前只能进入 `sourceMap`、`evidenceLog` 或 `warnings`。
- 使用 CLI 前先运行 `sifta-cli status`；失败、未认证或 provider 不可用时，不伪造成候选人结果。
- `find-people` 合格主路径默认 `--save`，把 run history 同步到 Web 供回看和复核；隐私硬停止、未认证、provider failure、0 人、schema 检查或用户要求不保存时用 `--no-save`。
- Web history 不是 shortlist、待触达或 pipeline 写入；只有用户显式要求加入业务流程，或在 Web 端执行对应动作后，才进入后续业务状态。
- 小批量 helper 完成、失败、认证不可用或返回 0 人后，本轮停止；不要追加网页、Exa、浏览器、第二个 helper 或第二次 CLI 凑人。
- 最终答复不输出命令、脚本名、events、timing、`target-count`、停止标记或使用了哪个 skill。

解析结构化结果时优先看 `people`、`searchStrategy`、`sourceMap`、`evidenceLog`、`crmExport` 和 `warnings`。warnings 必须转成招聘负责人能理解的还要确认、覆盖风险或下一步。

## 5. 用户可见报告

最终报告按 `templates/final-report.md` 写给招聘负责人，默认结构为：

```text
结论 → 人选和证据 → 其他线索 / 需要确认 → 下一步
```

0 人、provider failure 或 privacy hard-stop 时，不输出候选人表，改用 `结论 / 原因 / 下一步`。

内部仍要检查项目简报、来源地图、候选人分层、待核验线索、适配证明、覆盖缺口、停止原因、隐私边界和下一步，但不要把这些渲染成用户可见九个顶层标题。
