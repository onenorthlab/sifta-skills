# Sifta 最终报告模板（给用户看的版本）

用户是招聘负责人、猎头、投资人，不是工程师。最终回复用**招聘白话**，不要出现内部术语和工程过程：不写 `source-map lead / profile lead / candidate / verified / 证据等级 A/B/C / sourceFamily / checkpoint / functionCategory / priority`，也不写原始 JSON、命令、运行过程、用了哪个 skill、events、timing、target-count、bash/code block。除非用户明确问「怎么跑的」。

内部分级照常在脑子里走，但**渲染给用户时按下表翻译成白话**：

| 内部概念 | 用户看到的说法 |
| --- | --- |
| verified candidate / 证据等级 A | 可以聊（证据充分） |
| 待复核候选 / 证据等级 B | 建议先核实（方向对，差一两点确认） |
| source-map lead / profile lead / 证据等级 C | 待确认线索（还不算正式人选） |
| rejected | 已排除 |
| 全职候选 | 可直接联系 |
| 顾问 / 推荐人 | 适合做顾问或帮你引荐 |
| 产业标杆 | 行业标杆（参考、通常不直接招） |
| 适配证明包 | 为什么值得聊 |
| conversionBlocker / 弱项 | 还差什么 |
| nextVerification / 下一步核验 | 下一步怎么确认 |
| 覆盖风险 | 本轮覆盖缺口和不能下结论的地方 |

渲染模板使用固定标题。不要换成旧版四段式标题，也不要翻译成英文。

```markdown
**项目简报**
- 一两句话：本轮要找什么人、默认中国/中文生态优先是否生效、是否执行了实时搜索。

**来源地图**
| 来源 | 看什么 | 本轮怎么用 |
| --- | --- | --- |
| GitHub / LinkedIn / 学术图谱 / X / 产品页 | … | … |

**候选人分桶**
| 分桶 | 人选 | 现在做什么 | 为什么值得聊 | 把握 | 还要确认 | 链接 |
| --- | --- | --- | --- | --- | --- | --- |
| 可优先聊 | 张三 | 某公司 Agent 框架核心作者 | 主导 X 开源项目，和你要的 Y 高度对口 | 高 | 是否在看新机会 | [主页](url) |

> 把握：高=公开证据扎实、可优先联系；中=方向对、建议先核实一两点；低=只是线索、还需确认。

**待核验线索**（暂不算正式人选，需进一步确认才能算）
| 线索 | 为什么相关 | 还差什么 | 下一步怎么确认 |
| --- | --- | --- | --- |
| 某论文 / 某项目 / 某公司团队 | … | 还没确认是同一个人、缺个人主页 | … |

**适配证明包**
| 人选 / 线索 | 为什么符合需求 | 公开证据 | 把握 | 还要确认 | 下一步 |
| --- | --- | --- | --- | --- | --- |
| 张三 | … | … | 高 | … | … |

**覆盖风险**
- 比如：某渠道没返回结果、部分人只有弱线索、默认只看了中国 / 中文圈的人（只看公开职业信息，不靠姓名或照片判断）。

**停止条件**
- 本轮为什么停在这里：预算已用完 / 用户只要小批量 / 候选人需要人工核验 / 连接器失败不能换源凑数。

**执行合同**
- 本轮没有查询私人联系方式、没有自动发送消息；下一轮需要用户授权后再扩大来源、深挖个人资料或写触达草稿。

**下一步**
- 比如：要不要扩大范围再找一批 / 帮你写联系他们的招呼语 / 深挖某一位。
```

硬规则（即使说白话也不能破）：
- 候选人分桶里的每个人都要有「为什么值得聊」和「还要确认」，不能只给名字和链接。
- 链接用 `[主页](url)` / `[领英](url)` 形式，不输出裸 URL。
- 不编造邮箱、电话、薪资、是否想换工作、是否愿意搬迁。
- 没把握是不是同一个人时，写「可能是同一人，待确认」，不硬下结论。
- 论文 / 项目 / 公司团队这类只能进「待核验线索」，不能直接当候选人分桶。
