# Sifta 最终报告模板（给用户看的版本）

用户是招聘负责人、猎头、投资人，不是工程师。最终回复用**招聘白话**，不要出现内部术语和工程过程：不写 `sourceMapEntry / profile lead / candidate / verified / 证据等级 A/B/C / sourceFamily / checkpoint / functionCategory / priority`，也不写原始 JSON、命令、运行过程、用了哪个 skill、events、timing、target-count、bash/code block。除非用户明确问「怎么跑的」。

先回答招聘问题，再给技术证据：这个人值不值得约、适合按全职/顾问/推荐人哪种方式推进、第一通电话聊什么、主要顾虑是什么。repo、PR、commit、star、技术栈和项目功能只能作为支撑证据，不能写成主要卖点。

内部分级照常在脑子里走，但**渲染给用户时按下表翻译成白话**：

| 内部概念                                  | 用户看到的说法                     |
| ----------------------------------------- | ---------------------------------- |
| verified candidate / 证据等级 A           | 可以聊（证据充分）                 |
| reviewNeeded / 证据等级 B                 | 建议先核实（方向对，差一两点确认） |
| sourceMapEntry / profileLead / 证据等级 C | 待核验线索（还不算正式人选）       |
| rejected                                  | 已排除                             |
| 全职候选                                  | 可直接联系                         |
| 顾问 / 推荐人                             | 适合做顾问或帮你引荐               |
| 产业标杆                                  | 行业标杆（参考、通常不直接招）     |
| fitProofPacket                            | 为什么值得聊                       |
| conversionBlocker / 弱项                  | 还差什么                           |
| nextVerification / 下一步核验             | 下一步怎么确认                     |
| coverageGap                               | 覆盖风险和不能下结论的地方         |

渲染模板使用固定标题，顺序固定为：

```text
项目简报 → 来源地图 → 候选人分桶 → 待核验线索 → 适配证明包 → 覆盖风险 → 停止条件 → 执行合同 → 下一步
```

不要换成旧版四段式标题，不要使用「本轮目标 / 找人来源 / 推荐名单 / 匹配依据 / 本轮覆盖缺口」这组旧标题，也不要翻译成英文。

```markdown
**项目简报**

- 一两句话：本轮要找什么人、默认中国/中文生态优先是否生效、是否执行了实时搜索或只做计划。

**来源地图**
| 来源 | 看什么 | 本轮怎么用 | 状态 |
| --- | --- | --- |
| GitHub / LinkedIn / 学术图谱 / X / 产品页 | … | … | 已执行 / 未执行 / 阻塞 / 仅作线索 |

**候选人分桶**
| 分桶 | 人选 | 招聘判断 | 为什么值得聊 | 把握 | 招聘风险 | 下一步 | 链接 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 可优先聊 | 张三 | 先按平台型工程人选聊，确认是否愿意做执行环境和稳定性 | 公开资料显示他做过和你要的能力直接相关的开源实现 | 高 | 当前角色、可转化性和真实 ownership 还要确认 | 先补职业资料和 2-3 个核心贡献，再决定低压触达 | [主页](url) |

> 把握：高=公开证据扎实、可优先联系；中=方向对、建议先核实一两点；低=只是线索、还需确认。

**待核验线索**

暂不算正式人选，需进一步确认才能算。

| 线索                         | 为什么相关 | 还差什么                       | 下一步怎么确认 |
| ---------------------------- | ---------- | ------------------------------ | -------------- |
| 某论文 / 某项目 / 某公司团队 | …          | 还没确认是同一个人、缺个人主页 | …              |

**适配证明包**
| 人选 / 线索 | 需求匹配 | 公开证据 | 把握 | 主要风险 | 下一步 |
| --- | --- | --- | --- | --- | --- |
| 张三 | 可优先核验是否适合全职/顾问推进 | … | 高 | … | … |

**覆盖风险**

- 比如：某渠道没返回结果、部分人只有弱线索、默认只看了中国 / 中文圈的人（只看公开职业信息，不靠姓名或照片判断）。

**停止条件**

- 本轮为什么停在这里：预算已用完 / 用户只要小批量 / 候选人需要人工核验 / 连接器失败不能换源凑数。

**执行合同**

- 本轮没有查询私人联系方式、没有自动发送消息；下一轮需要用户授权后再扩大来源、深挖个人资料或写触达草稿。

**下一步**

- 比如：要不要扩大范围再找一批 / 帮你写联系他们的招呼语 / 深挖某一位。
```

硬规则（即使说白话也不能破）：

- 候选人分桶里的每个人都要有「为什么值得聊」和「还要确认」，不能只给名字和链接。
- 链接用 `[主页](url)` / `[领英](url)` 形式，不输出裸 URL。
- 不编造邮箱、电话、薪资、是否想换工作、是否愿意搬迁。
- 没把握是不是同一个人时，写「可能是同一人，待确认」，不硬下结论。
- 论文 / 项目 / 公司团队这类只能进「待核验线索」，不能直接当候选人分桶。
- 默认中国/中文生态人才池未被用户放宽时，缺公开中国/中文生态职业信号的人只能进「待核验线索 / 全球备选」，不能进「候选人分桶」。
- 候选人分桶先回答招聘问题：这个人现在值不值得聊、适合全职/顾问/推荐人/待核验中的哪一种、主要风险是什么；技术证据只放在支撑列里。
- 「招聘判断」只写动作和定位，例如“可优先聊”“建议先核实”“适合做顾问或推荐人”，不要写 sandbox / MCP / message bus 等技术清单。
- 「为什么值得聊」控制在一句人话，说明和岗位最相关的一条理由；技术细节超过一个就移到「适配证明包」。
- 「下一步」必须是招聘动作，例如“补职业资料”“核 2-3 个核心 PR”“低压约技术预核验”，不要写成长篇技术尽调清单。
- 候选人分桶的「角色 / title」必须能在所列链接的原文里找到原话；**禁止为贴合岗位需求而改写、拔高或美化 title**。原文 title 与岗位方向不同时，如实写原文 title，再用一句话说明「和岗位的契合点在哪」，不要把契合点写进 title。
- 每个推荐人选都要显式回答「怎么联系到」（有公开职业渠道 / 需引荐 / 暂无）和「可招聘性的结构信号」（在读≈可动性高 / 在职大厂≈需撬动 / 自营创业≈低 / 项目或公司刚结束≈可能在看）。只标公开可见的结构信号，能力强不等于可招。
- 「是否想换工作 / 求职意愿」是公开信息的天花板，**不判断、不假装、不靠 hireable 标记或猜测**；把它转成触达前要问清的 1-2 个问题，写进「下一步」（例如「先确认当前方向和是否开放外部机会」）。
- 自营创业中、在营公司核心创始人默认按顾问 / 引荐 / 标杆处理，不包装成可直接全职聊。
