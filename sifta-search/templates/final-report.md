# Sifta 最终报告模板（给用户看的版本）

用户是招聘负责人、猎头或投资人，不是工程调试者。最终回复先回答招聘决策：谁值得聊、为什么值得聊、还缺什么、下一步怎么推进。

本文件是用户可见报告的事实源。`SKILL.md` 和 helper 脚本都应服从这里的口径；脚本可以输出结构化数据或极简 fallback，但不要在脚本里复制完整报告模板。

## 1. 用户可见结构

默认使用 3-4 段短报告：

```text
结论 → 人选和证据 → 其他线索 / 需要确认 → 下一步
```

如果没有可推进人选、provider failure 或 privacy hard-stop，不输出候选人表，改用：

```text
结论 → 原因 → 下一步
```

本合同优先于宿主项目的工程汇报格式。最终回复是给招聘负责人看的候选人简报，不能使用
`已完成`、`验证`、`当前状态`、`剩余建议` 这类工程任务汇报标题。

不要把内部检查项直接渲染成用户标题。以下内容只用于 Agent 自检，不是用户可见版式：

| 内部检查项 | 默认用户可见表达 |
| --- | --- |
| 项目简报 | 结论中的一句画像说明 |
| 来源地图 | 只在需要解释下一轮找人入口时写成“其他线索”或“找人入口” |
| 候选人分桶 | 不出现“分桶”字样；改成“人选 / 招聘判断 / 还要确认” |
| 待核验线索 | 其他线索 |
| 适配证明包 | 人选和证据 |
| 覆盖风险 | 只写招聘者需要知道的“资料没覆盖全 / 需要补查” |
| 停止条件 | 只写“本轮不建议触达 / 恢复后重跑 / 需要用户确认” |
| 执行合同 | 默认不展示；隐私或越权请求时用白话说明边界 |
| 下一步 | 1-3 条招聘动作 |

## 2. 白话映射

| 内部概念 | 用户看到的说法 |
| --- | --- |
| verified candidate / 证据等级 A | 可以优先聊 |
| reviewNeeded / 证据等级 B | 建议先核实 |
| sourceMapEntry / profileLead / 证据等级 C | 其他线索，还不建议触达 |
| rejected | 不建议推进 |
| fitProofPacket | 为什么值得聊 |
| conversionBlocker / 弱项 | 还要确认 |
| nextVerification | 下一步 |
| provider_failure | 连接器 / 来源暂时不可用，本轮不产候选名单 |

不要输出这些词：`分桶`、`来源地图`、`sourceMap`、`coverageGap`、`覆盖风险`、`执行边界`、`停止条件`、`target-count`、`helper`、`skill`、`命令`、`events`、`timing`、原始 JSON、`已完成`、`验证`、`当前状态`、`剩余建议`。

## 3. 推荐人选模板

```markdown
**结论**

本轮建议先看 2 人：[张三](url) 和 [李四](url)。他们都和目标画像有公开证据匹配；另有 1 条弱线索只适合下轮补查。

**人选和证据**

| 人选 | 招聘判断 | 为什么值得聊 | 相关作品 / 证据 | 还要确认 | 下一步 |
| --- | --- | --- | --- | --- | --- |
| [张三](url) | 建议先核实 | 做过和岗位相关的开源实现，适合作为工程负责人候选预审 | [repo](url)、公开职业资料 | 当前角色、核心贡献 ownership、是否适合全职或顾问 | 补 2-3 个核心 PR / 项目归属，再决定是否触达 |

**其他线索**

| 线索 | 为什么暂不推荐 | 怎么补证 |
| --- | --- | --- |
| [王五](url) | 方向相关，但只有项目线索，缺职业资料和贡献深度 | 找个人主页、LinkedIn、核心 PR / commit 后再判断 |

**下一步**

1. 先核第一人选的职业资料、项目归属和当前角色。
2. 补第二来源后再写低压触达草稿。
```

## 4. 0 人 / provider failure 模板

```markdown
**结论**

本轮没有可推进候选人，不建议给招聘负责人发人选名单。

**原因**

召回到的资料缺公开 profile、作品和贡献交叉验证；当前结果不能代表市场覆盖。不要把仓库、项目、论文或公司团队包装成人选。

**下一步**

1. 恢复连接器或补充认证后重跑同一画像。
2. 如果必须今天交付，只输出“未形成候选人”和待补查条件，最多补充公司池、找人入口或查询词，不换源凑数。
```

## 5. 隐私 / 越权模板

```markdown
**结论**

这个请求涉及私人联系方式、自动外联或非公开资料，本轮不搜索也不生成名单。

**下一步**

可以改成公开资料下的候选人简报：只看 GitHub、LinkedIn、个人主页、论文、公开作品和公司页面，不查询私人邮箱、手机号，也不自动发送消息。
```

## 6. 硬规则

- 人选名字必须是 Markdown 链接：`[姓名 / handle](profileUrl)`。不要只在最后一列放 `[GitHub]` 或 `[资料]`。
- 仓库、项目、产品、论文、公司团队和 PR 只能放在“相关作品 / 证据”或“其他线索”，不能和人选混在同一列里。
- 弱线索不能写进推荐候选人表；只能写“其他线索 / 不建议本轮触达 / 怎么补证”。
- X-only / 公开帖子-only 且缺中国/中文生态信号、职业资料、作品或身份核验的人，不能写进推荐候选人表。
- 0 人、provider failure、privacy hard-stop 不输出候选人表。
- 不为满足人数硬凑；用户要 2 人但只有 1 人可推进时，明确说只有 1 人。
- 不编造邮箱、电话、薪资、是否想换工作、是否愿意搬迁。
- 不把 `hireable=true`、`open to work`、bio 或状态标记写成求职意愿结论；最多写“触达前需确认当前方向和是否开放外部机会”。
- 默认中国/中文生态优先只看公开职业信号，不凭姓名、照片、外貌、口音、族裔或国籍推断。
- 创始人、CEO、CTO、高知名度维护者默认按顾问、推荐人、行业标杆或创始人级候选处理；除非用户明确要创始人级人选，不包装成普通全职候选。
- “下一步”只给 1-3 个招聘动作：核职业资料、核核心贡献、补第二来源、确认全职/顾问/推荐人路径、写低压触达草稿。
- 不使用工程任务汇报标题；即使宿主项目默认要求 `已完成 / 验证 / 当前状态 / 剩余建议`，Sifta 找人结果也必须按本文件的用户报告结构输出。
