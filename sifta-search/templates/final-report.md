# Sifta 最终报告模板（给用户看的版本）

用户是招聘负责人、猎头、投资人，不是工程师。最终回复用**招聘白话**，不要出现内部术语和工程过程：不写 `sourceMapEntry / profile lead / candidate / verified / 证据等级 A/B/C / sourceFamily / checkpoint / functionCategory / priority`，也不写原始 JSON、命令、运行过程、用了哪个 skill、events、timing、target-count、bash/code block。除非用户明确问「怎么跑的」。

先回答招聘问题，再给公开证据：这个人值不值得约、适合按全职/顾问/推荐人哪种方式推进、第一通电话聊什么、主要顾虑是什么。repo、PR、commit、star、技术栈和项目功能只能作为支撑证据，不能写成主要卖点。

## 1. 用户可见结构

用户可见报告固定为 4 段决策简报：

```text
结论 → 人选和证据 → 风险和待核验 → 下一步
```

不要把内部检查项直接渲染成九个大标题，也不要让招聘负责人读九段报告。九项不是用户可见版式、段落数或长度要求，只是 Agent 内部防漏检查：项目简报、来源地图、候选人分桶、待核验线索、适配证明包、覆盖风险、停止条件、执行合同、下一步。它们分别折进 4 段：

| 内部检查项 | 用户可见位置              |
| ---------- | ------------------------- |
| 项目简报   | 结论                      |
| 来源地图   | 人选和证据 / 风险和待核验 |
| 候选人分桶 | 人选和证据                |
| 待核验线索 | 风险和待核验              |
| 适配证明包 | 人选和证据                |
| 覆盖风险   | 风险和待核验              |
| 停止条件   | 风险和待核验              |
| 执行合同   | 风险和待核验              |
| 下一步     | 下一步                    |

## 2. 白话映射

内部分级照常在脑子里走，但渲染给用户时按下表翻译成白话：

| 内部概念                                  | 用户看到的说法                     |
| ----------------------------------------- | ---------------------------------- |
| verified candidate / 证据等级 A           | 可以聊（证据充分）                 |
| reviewNeeded / 证据等级 B                 | 建议先核实（方向对，差一两点确认） |
| sourceMapEntry / profileLead / 证据等级 C | 待核验线索（还不算正式人选）       |
| rejected                                  | 已排除                             |
| 全职候选                                  | 可直接联系                         |
| 顾问 / 推荐人                             | 适合做顾问或帮你引荐               |
| 产业标杆                                  | 行业标杆（参考、通常不直接招）     |
| fitProofPacket                            | 为什么值得聊                       |
| conversionBlocker / 弱项                  | 还差什么                           |
| nextVerification / 下一步核验             | 下一步怎么确认                     |
| coverageGap                               | 覆盖风险和不能下结论的地方         |

## 3. 模板

```markdown
**结论**

- 一两句话：本轮能不能推进、最值得先看谁、默认中国/中文生态优先是否生效、有没有执行实时搜索。
- 如果没有可推进人选，直接说「本轮不建议推进候选人」，不要用来源线索凑数。

**人选和证据**

| 分桶       | 人选 / 线索 | 招聘判断                                        | 为什么值得聊                           | 把握 | 证据来源                  | 下一步                      | 链接          |
| ---------- | ----------- | ----------------------------------------------- | -------------------------------------- | ---- | ------------------------- | --------------------------- | ------------- |
| 建议先核实 | 张三        | 先按平台型工程人选聊，确认是否适合全职/顾问推进 | 公开资料显示他做过和岗位相关的开源实现 | 中   | GitHub profile + 相关项目 | 补职业资料和 2-3 个核心贡献 | [GitHub](url) |

> 把握：高=公开证据扎实、可优先联系；中=方向对、建议先核实一两点；低=只是线索、还需确认。

**风险和待核验**

| 类型       | 内容                                                             | 为什么重要             | 怎么确认                                |
| ---------- | ---------------------------------------------------------------- | ---------------------- | --------------------------------------- |
| 待核验线索 | 某论文 / 某项目 / 某公司团队                                     | 方向相关但还不是候选人 | 找个人主页、LinkedIn、GitHub 或公司 bio |
| 覆盖风险   | 某渠道没有返回、部分人只有弱线索                                 | 影响本轮结论置信度     | 下一轮补指定来源                        |
| 停止原因   | 用户只要小批量 / 连接器失败 / 需要人工确认                       | 不能换源凑数           | 用户确认后再扩展                        |
| 执行边界   | 不查询私人邮箱、手机号、非公开联系方式，不自动发送消息或提交表单 | 避免越界和误导         | 获得授权后再写触达草稿                  |

**下一步**

- 只给 1-3 个具体动作：核某人的职业资料、补第二来源、扩大同来源召回、写低压触达草稿。
```

## 4. 硬规则

- 候选人分桶里的每个人都要有「为什么值得聊」和「还要确认」，不能只给名字和链接。
- 链接用 `[主页](url)` / `[领英](url)` / `[GitHub](url)` 形式，不输出裸 URL。
- 如果 helper、草稿或来源表里已经有 `[资料](url)` / `[主页](url)` / `[GitHub](url)`，最终答复必须保留 Markdown 链接格式；不要改成裸 URL 或单独一行 URL。
- 不编造邮箱、电话、薪资、是否想换工作、是否愿意搬迁。
- 没把握是不是同一个人时，写「可能是同一人，待确认」，不硬下结论。
- 论文 / 项目 / 公司团队这类只能进「风险和待核验」，不能直接当正式候选人。
- 默认中国/中文生态人才池未被用户放宽时，缺公开中国/中文生态职业信号的人只能进「待核验线索 / 全球备选」，不能进可推进候选。
- 候选人分桶先回答招聘问题：这个人现在值不值得聊、适合全职/顾问/推荐人/待核验中的哪一种、主要风险是什么；技术证据只放在支撑列里。
- 「招聘判断」只写动作和定位，例如“可优先聊”“建议先核实”“适合做顾问或推荐人”，不要写 sandbox / MCP / message bus 等技术清单。
- 「为什么值得聊」控制在一句人话，说明和岗位最相关的一条理由；技术细节超过一个就移到证据来源或风险表。
- 「下一步」必须是招聘动作，例如“补职业资料”“核 2-3 个核心 PR”“低压约技术预核验”，不要写成长篇技术尽调清单。
- 候选人分桶的「角色 / title」必须能在所列链接的原文里找到原话；**禁止为贴合岗位需求而改写、拔高或美化 title**。原文 title 与岗位方向不同时，如实写原文 title，再用一句话说明「和岗位的契合点在哪」，不要把契合点写进 title。
- 候选人分桶里的每个人都要显式回答「怎么联系到」（有公开职业渠道 / 需引荐 / 暂无）和「可招聘性的结构信号」（在读≈可动性高 / 在职大厂≈需撬动 / 自营创业≈低 / 项目或公司刚结束≈可能在看）。只标公开可见的结构信号，能力强不等于可招。
- 「是否想换工作 / 求职意愿」是公开信息的天花板，**不判断、不假装、不靠 hireable 标记或猜测**；把它转成触达前要问清的 1-2 个问题，写进「下一步」（例如「先确认当前方向和是否开放外部机会」）。
- 不要把 `hireable=true`、`Open to new opportunities`、`open to work`、profile bio 或状态标记写成“可招聘性”或“求职意愿”结论；最多写“公开 profile 有开放机会相关表述，触达前必须确认当前方向和是否开放外部机会”。
- 自营创业中、在营公司核心创始人默认按顾问 / 引荐 / 标杆处理，不包装成可直接全职聊。
